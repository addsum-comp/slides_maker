# slide-maker — 設計・運用ガイド

このドキュメントは、本スキルがどのように構築されているか、そしてそれをどう捉えるべきかを、ユーザー（あなた）と保守担当者の双方に向けて説明するものです。スキルを*実行する*モデルが読むのは `SKILL.md` と `references/` であり、この README はそれらを俯瞰する地図にあたります。

---

## 1. これは何か

あらゆる聴衆、あらゆる言語に対応し、テンプレートの有無や元素材の有無を問わず、プレゼンテーション品質の `.pptx` デッキを構築・再設計・批評するスキルです。その唯一の信念は、**スライドは話者のための視覚的補助であって、読ませる文書ではない**ということ。だからこそ、あらゆる判断は「数秒で理解できること」を最適化の基準とします。

本スキルは意図的に**インタビュー優先**かつ**批評ゲート方式**で設計されています。前提を置く前にまず問いかけ、自らの出力を信頼しません。デッキが「完成」とみなされる前には、独立した批評者の同意が必要です。

---

## 2. コアパイプライン（自動モード）

すべてのデッキは7つのステップを通って仕上がります（正式な仕様は `SKILL.md`）。

| ステップ | 何が起きるか | なぜ存在するか |
|---|---|---|
| **0 — インタビュー** | 1回の `AskUserQuestion` バッチで、テンプレート、目的と聴衆、元素材、スタイルを確認（＋フォローアップ：カンファレンス会場、新規テンプレート）。 | ユーザーの要件こそが真実の源。要件は*学び取る*ものであり、過去のデッキから引き継ぐものではありません。 |
| **1 — 理解** | すべての元素材を深く読み込み、**理解ブリーフ**を作成（一文のメッセージ、貢献、手法の本質、各図表が*何のためにあるか*、限界）。 | 見た目は正しくても内容を読み違えたデッキは、専門家を欺けません。忠実性はここから始まります。 |
| **2 — キャンバス** | 出力フォルダ（`~/Downloads/<deck>/`）を決定し、テンプレートを読み込むか、目的に合った見た目を設計する。パレットとフォント（CJK の `EAFONT` を含む）を設定。 | ブランディングはレイアウトに宿ります。設計は、一語も読まれる前に、その文書が*どういう種類のもの*かを伝えるべきです。 |
| **3 — プラン** | スライド枚数は時間予算に比例（約1枚/分）。短い講演で約6〜9枚、長い講演・講義・学位審査・ジョブトークで約10〜20枚以上。各スライドは1つのアイデア、テイクアウェイ優先、目的に合わせた弧を描く。約15枚以上ならセクションのファンアウト（ステップ4）へ。 | アウトラインの修正は安く、完成したデッキの修正は高くつきます。 |
| **4 — ビルド** | `deckkit` ヘルパーを使った1本のビルドスクリプト。元の図をそのまま使う、ガター、ローテーションするアクセント、本物の数式、単一言語、任意のビルド/アニメーション、スピーカーノート。 | python-pptx は高速。1回のスクリプト実行で、一貫した一人の作者によるデッキになります。 |
| **5 — レンダー＋批評ループ** | PNG にレンダリングして*見る*。次に**独立した批評サブエージェント**が JSON を返す（同意 / 修正＋スライドごとの修正点）。同意が得られるまでループ。 | python-pptx は目隠しで書き出します。オーバーフロー・コントラスト・グリフのバグはピクセルでしか現れません。自らの作品の審判を自ら務めることはできません。 |
| **6 — 引き渡し＋反復** | ユーザーに見せ、フォルダパスを伝え、編集可能性と2つの変更レーンを説明し、フィードバックを取り込む。 | デッキはユーザーのものであり、安全に調整し続けられます。 |

**アクター・批評ループこそが品質エンジンです。** その*重み*は重要度に応じてスケールしますが（ラボミーティングなら批評者1名、カンファレンス/学位審査/ピッチなら異なる観点を持つ2〜3名の並列パネル）、ループそのものは譲れません。

## 3. 2つのモード

- **自動（デフォルト）：** インタビュー → ビルド → 高い基準に達するまでの批評ループ → 提示。批評者が*品質*を担保します。
- **協働（オプトイン）：** 安価な**承認ゲート**を追加します。*方向性*を選び（実際にレンダリングされた2〜3枚のアーキタイプスライド）→ *アウトライン*を承認 → 残りをビルド。ゲートは批評者には読み取れない*嗜好*（テイスト）を捉えます。土台のエンジンは同じで、承認が加わるだけです。（`references/collaborative-mode.md`、`scripts/archetypes.py`）

## 4. シナリオマップ — リクエストがどの経路をたどるか

インタビュー（ステップ0、特にQ3）がリクエストを振り分けます。

| ユーザーが望むこと… | 経路 |
|---|---|
| 自分のコード/論文/ドキュメントからのデッキ | ビルド経路（ステップ1〜6）、コンテンツ分岐 |
| 素材なしのデッキ | ビルド経路。専門知識から下書きし、Web 検索で裏付け、アウトラインを確認 |
| **自分のデッキを改善**したい | **再設計経路** — まず診断し、スコープを確認し、ユーザーのコンテンツ/図を再利用して再構築（`references/redesign-existing-deck.md`） |
| **例のような見た目**のデッキ | スタイル模倣 — スタイルブリーフを書き、その見た目を再現（`references/style-analysis.md`） |
| **カンファレンス**講演 | 会場を特定して Web 調査し（ルール、テンプレート、聴衆）、それに合わせてビルド |
| **ポスター** | 限定対応：単一の大型キャンバス。制作の原則は通用するが、本スキルは講演向けにチューニングされているため、まず仕様を確認 |
| **非英語 / CJK** のデッキ | `EAFONT` を設定し、単一言語の規律、CJK タイポグラフィ（`references/multilingual.md`） |
| **大規模**デッキ（15枚以上） | 任意のセクションファンアウト：共有 `style.py`、並列セクション著者、`assemble.py`、批評パネル（`references/large-deck-orchestration.md`） |
| **まず選択肢を見たい** | 協働モードのゲート |
| **引き渡し後の変更** | 安全に反復 — 手作業の編集を決して上書きしない（`references/handoff-and-iteration.md`） |

## 5. ファイルマップ

**背骨**
- `SKILL.md` — モデルが従う運用指示（ステップ0〜6、ルール）。

**エンジン（`scripts/`）**
- `deckkit.py` — ビルドキット：テキスト/シェイプ/コンポーネントのヘルパー（`bullet`、`callout`、`chip`、`arrow`、`modbox`、`hrule`）、数式（`eq_par`、`equation_png`）、`speaker_notes`、コントラストチェック、パレット/フォント（CJK の `EAFONT` を含む）、テンプレート再利用（`open_template`、`content_slide`）、テンプレートなしのクローム（`blank_deck`、`title_bar`、`footer`）。インポートして使うこと。プリミティブを再実装しないこと。
- `render_deck.sh` — `.pptx` → スライドごとに1枚の PNG（LibreOffice → PDF → PNG）。クロスプラットフォーム。プライベートな LibreOffice プロファイルを使うため、並列/共存するレンダリングが衝突しません。
- `check_env.sh` — ツールチェーンの一度きりのプリフライト。
- `anim.py` — python-pptx が書けない PowerPoint のビルド/アニメーションのタイミング XML を注入。
- `assemble.py` — 並列で著作されたセクションモジュールを1つのデッキに結合（壊れやすいマージなしで）。
- `archetypes.py` — 協働ゲート向けに、方向性ごとに同じプレビュースライドをビルド。
- `inspect_template.py` — テンプレートのレイアウト/プレースホルダ/ロゴを出力。
- `extract_deck.py` — 既存デッキからテキスト/表/図を*抜き出す*（再設計＋突き合わせ）。
- `export_notes.py` — デッキのスピーカーノートをプレーンテキストのリハーサル台本にエクスポート。

**判断**
- `agents/critic.md` — 独立した批評者のブリーフ＋ JSON スキーマ。
- `references/review-rubrics.md` — 汎用ルーブリック＋目的別オーバーレイ（研究に基づく）。
- `references/design-principles.md` — その技法と「なぜ」。

**シナリオ別リファレンス**
- `design-by-purpose.md` · `animation.md` · `multilingual.md` · `font-guidance.md` ·
  `style-analysis.md` · `redesign-existing-deck.md` · `collaborative-mode.md` ·
  `large-deck-orchestration.md` · `handoff-and-iteration.md`
- `examples/` — 完成したビルドスクリプト、共有スタイル＋セクションモジュールの規約。

**外部（スキルの一部ではない）**
- `~/.claude/slide-templates/` — ユーザー個人のテンプレートレジストリ。選択肢のために読み込み、新しいプロファイルをここに書き込む。新規ユーザーでは空。

## 6. スキルに組み込まれた設計原則

1. **アーティファクトよりも要件。** テンプレート、古いデッキ、モデルのテイストはいずれも*入力*であって指示ではありません。それらが明示された要件と衝突したときは、要件が勝ちます。
2. **厳格な忠実性。** あらゆる主張/数値/図は元素材にまで遡れること。唯一の例外は、明確に表示された将来志向のコンテンツです。
3. **独立した批評。** 別のエージェントがレンダリングされたピクセルを審査します。その独立性こそが「同意」を意味あるものにします。
4. **収集は並列化し、理解は決して並列化しない。** 読み込みと素材準備はファンアウトし、一筋の論旨は一人の頭が保持します。
5. **目的に合った設計。** 学位審査、経営層への報告、講義が同じ見た目であってはなりません。
6. **単一言語を最後まで貫く。**
7. **スクリプトが真実の源であり、`.pptx` はアーティファクトである。** 再現可能で、ユーザーの編集を失うことなく安全に反復できます。

---

## 7. 既知の限界（これらは正直に伝えること）

- **テキストの高さは測定ではなく推定。** python-pptx はレンダリング後のテキストの高さを知り得ないため、`bullet`/`callout` の高さはスケールした推定値です。オーバーフローはレンダーループ（ステップ5）で捕捉します。必ず PNG を見ること。
- **アニメーションは静的にプレビューできない。** レンダーは最終的なビルド状態のみを示します。ビルドの*順序*は実際の PowerPoint で検証し、引き渡し時にユーザーへ説明します。
- **RTL スクリプト（アラビア語/ヘブライ語）** は既知の弱点です — 双方向の再フローはありません。
- **ポスター** は最小限のサポートのみ。本スキルは講演向けにチューニングされています。
- **フォントは埋め込まれない**（python-pptx の制約）— 非標準/CJK フォントへの依存は引き渡し時に明示します。

## 8. ツールチェーン

`python-pptx`、`pymupdf`（レンダリング）、`matplotlib` ＋ `Pillow`（数式/チャート）、そしてレンダリング用の LibreOffice（`soffice`）。新しいマシンでは一度 `bash scripts/check_env.sh` を実行してください。不足しているものに対する正確な対処法を出力します。
