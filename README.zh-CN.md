# slide-maker — 设计与操作指南

本文档说明这个 skill 是如何构建的，以及应当如何理解它——既面向用户（也就是你），也面向任何维护它的人。真正*运行*该 skill 的模型读取的是 `SKILL.md` 和 `references/`；本 README 则是位于它们之上的总览地图。

---

## 1. 它是什么

一个用于构建、重新设计并评审演示级 `.pptx` 幻灯片的 skill，适用于任何受众、任何语言，可以有模板也可以没有，可以有源材料也可以没有。它只有一个核心信念：**幻灯片是给演讲者用的视觉辅助，而不是一份供人阅读的文档**——因此每一个决策都围绕"几秒钟内被理解"来优化。

它刻意做到**先访谈**且**受评审把关**：在做出假设之前先发问，并且不信任自己的输出——必须有一位独立评审者认可，一份幻灯片才算"完成"。

---

## 2. 核心流程（自动模式）

每一份幻灯片都要经过七个步骤（`SKILL.md` 是权威规范）：

| 步骤 | 发生了什么 | 为何存在 |
|---|---|---|
| **0 — 访谈** | 一次 `AskUserQuestion` 批量提问：模板、目的与受众、源材料、风格。（外加追问：会议场合、新模板。） | 用户的需求才是事实来源；你要去*了解*它们，而绝不从此前的某份幻灯片中继承。 |
| **1 — 理解** | 深入通读所有源材料；写出一份**理解纲要**（一句话核心信息、贡献点、方法精髓、每张图/表各自*用来做什么*、局限）。 | 一份看起来对、却误读了工作的幻灯片骗不过任何专家。忠实性从这里开始。 |
| **2 — 画布** | 确定输出文件夹（`~/Downloads/<deck>/`），加载模板*或*设计一套契合目的的外观；设定配色/字体（含 CJK `EAFONT`）。 | 品牌元素存在于版式中；设计应当在读到一个字之前就传达出这是哪*一类*文档。 |
| **3 — 规划** | 幻灯片数量按时间预算缩放（约 1 张/分钟）：短报告约 6–9 张，较长报告/讲课/答辩/求职报告约 10–20+ 张。每张一个想法、结论先行、整体弧线契合目的；约 15+ 张时进行章节扇出（步骤 4）。 | 修改大纲很便宜；修改一份成稿的幻灯片很昂贵。 |
| **4 — 构建** | 用 `deckkit` 辅助函数写一个构建脚本。整张源图、留白、轮换的强调色、真实公式、单一语言、可选的逐步呈现/动画、演讲备注。 | python-pptx 很快；一次脚本运行，一位连贯的作者。 |
| **5 — 渲染 + 评审循环** | 渲染成 PNG 并*亲眼看*；然后由一个**独立评审子代理**返回 JSON（认可 / 修订 + 逐张修复项）。循环直到认可为止。 | python-pptx 是盲写的——溢出、对比度、字形等问题只有在像素层面才会显现。你不能当自己作品的裁判。 |
| **6 — 交付 + 迭代** | 展示给用户，给出文件夹路径，说明可编辑性以及两条变更通道，并纳入反馈。 | 这份幻灯片归用户所有，他们可以持续地、安全地调整。 |

**演员—评审（actor–critic）循环是质量引擎。** 它的*分量*随重要程度缩放（实验室例会用一位评审；会议/答辩/路演则用 2–3 位带不同视角的评审并行组成评审团），但循环本身不可妥协。

---

## 3. 两种模式

- **自动模式（默认）：** 访谈 → 构建 → 评审循环至高标准 → 展示。评审者把控的是*质量*。
- **协作模式（按需开启）：** 增加低成本的**审批关卡**——先挑选一个*方向*（2–3 张真实渲染出来的原型幻灯片）→ 批准*大纲* → 再构建其余部分。这些关卡把控的是*偏好*（品味），而这是评审者读不出来的。底层是同一个引擎；它只是增加了几道审批。（`references/collaborative-mode.md`、`scripts/archetypes.py`。）

---

## 4. 场景地图——一个请求会走哪条路径

访谈（步骤 0，尤其是 Q3）负责为请求选路：

| 用户想要…… | 路径 |
|---|---|
| 从自己的代码/论文/文档做一份幻灯片 | 构建路径（步骤 1–6），内容分支 |
| 一份没有材料的幻灯片 | 构建路径；基于专业知识起草，并用网络检索来夯实依据，确认大纲 |
| **改进他们自己**的幻灯片 | **重设计路径**——先诊断，确认范围，复用其内容/图表进行重建（`references/redesign-existing-deck.md`） |
| 一份**看起来像某个示例**的幻灯片 | 风格模仿——写一份风格纲要，复现其外观（`references/style-analysis.md`） |
| 一场**会议**报告 | 识别并网络调研该场合（规则、模板、受众），然后据此构建 |
| 一张**海报** | 限定范围：单张大画布；制作原则依然成立，但本 skill 是为报告调优的——先确认规格 |
| 一份**非英语 / CJK** 幻灯片 | 设定 `EAFONT`、坚持单一语言、采用 CJK 排版（`references/multilingual.md`） |
| 一份**大型**幻灯片（15+ 张） | 可选的章节扇出：共享 `style.py`、并行的章节作者、`assemble.py`、评审团（`references/large-deck-orchestration.md`） |
| **先看若干选项** | 协作模式关卡 |
| **交付之后再做变更** | 安全迭代——绝不覆盖手工编辑（`references/handoff-and-iteration.md`） |

## 5. 文件地图

**主干**
- `SKILL.md` — 模型所遵循的操作指令（步骤 0–6，以及各项规则）。

**引擎（`scripts/`）**
- `deckkit.py` — 构建工具箱：文本/形状/组件辅助函数（`bullet`、`callout`、`chip`、`arrow`、`modbox`、`hrule`），公式（`eq_par`、`equation_png`），`speaker_notes`，对比度检查，配色/字体（含 CJK `EAFONT`），模板复用（`open_template`、`content_slide`）以及无模板版式装饰（`blank_deck`、`title_bar`、`footer`）。直接 import 它；不要重新造原语。
- `render_deck.sh` — `.pptx` → 每张幻灯片一张 PNG（LibreOffice → PDF → PNG）。跨平台；使用一个私有 LibreOffice 配置文件，以便并行/共存的渲染互不冲突。
- `check_env.sh` — 对工具链做一次性的预检。
- `anim.py` — 注入 python-pptx 无法写入的 PowerPoint 逐步呈现/动画计时 XML。
- `assemble.py` — 将并行编写的章节模块合并成一份幻灯片（不依赖脆弱的合并操作）。
- `archetypes.py` — 为协作关卡按方向构建相同的预览幻灯片。
- `inspect_template.py` — 打印某个模板的版式/占位符/徽标。
- `extract_deck.py` — 从一份现有幻灯片中*抽出*文本/表格/图表（重设计 + 核对）。
- `export_notes.py` — 将一份幻灯片的演讲备注导出为纯文本排练稿。

**裁判**
- `agents/critic.md` — 独立评审者的任务说明 + JSON schema。
- `references/review-rubrics.md` — 通用评分标准 + 按目的的叠加项（基于研究依据）。
- `references/design-principles.md` — 手艺及其"为什么"。

**按场景的参考文件**
- `design-by-purpose.md` · `animation.md` · `multilingual.md` · `font-guidance.md` ·
  `style-analysis.md` · `redesign-existing-deck.md` · `collaborative-mode.md` ·
  `large-deck-orchestration.md` · `handoff-and-iteration.md`
- `examples/` — 一个完整的构建脚本范例，以及共享风格 + 章节模块的约定。

**外部（不属于本 skill）**
- `~/.claude/slide-templates/` — 用户的个人模板库；读取以供选择，并将新的配置写入其中。对新用户而言为空。

## 6. 内置于本 skill 的设计原则

1. **需求高于既有产物。** 一个模板、一份旧幻灯片，或者模型自身的品味都是*输入*，而非指令。当它们与已陈述的需求相冲突时，以需求为准。
2. **严格忠实。** 每一项主张/数字/图表都可追溯到源材料。唯一的例外是经过明确标注的前瞻性内容。
3. **独立评审。** 由一个独立的代理来评判渲染出的像素——正是它的独立性，才让"认可"具有意义。
4. **可以并行收集，绝不并行理解。** 把阅读/素材准备扇出去；但要有一个头脑把握整条主线。
5. **契合目的的设计。** 一场答辩、一份高管汇报和一堂讲课不应当看起来一样。
6. **单一语言，自始至终保持一致。**
7. **脚本是事实来源；`.pptx` 只是产物。** 可复现，并且在迭代时能安全地不丢失用户的编辑。

---

## 7. 已知局限（要对这些坦诚）

- **文本高度是估算的，不是测量的。** python-pptx 无法得知渲染后的文本高度，所以 `bullet`/`callout` 的高度是按比例估算出来的——渲染循环（步骤 5）正是用来捕捉溢出的手段。务必亲眼看 PNG。
- **动画无法静态预览。** 渲染只能呈现构建后的最终状态；构建的*顺序*要在真实的 PowerPoint 中验证，并在交付时向用户说明。
- **RTL 文字（阿拉伯语/希伯来语）** 是一个已知的薄弱环节——没有双向（bidi）重排。
- **海报** 仅有最低限度的支持；本 skill 是为报告调优的。
- **字体不会被嵌入**（python-pptx 的限制）——在交付时标注出任何非标准/CJK 字体依赖。

---

## 8. 工具链

`python-pptx`、`pymupdf`（渲染）、`matplotlib` + `Pillow`（公式/图表），以及用于渲染的 LibreOffice（`soffice`）。在一台新机器上运行一次 `bash scripts/check_env.sh`；对于任何缺失项，它都会打印出确切的修复方法。
