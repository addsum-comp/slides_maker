# slide-maker — 설계 및 운영 가이드

이 문서는 스킬이 어떻게 구성되어 있고 어떻게 그 동작을 이해해야 하는지를 설명합니다 — 사용자
(당신)와 이를 유지보수하는 모든 사람을 위한 것입니다. 스킬을 *실행하는* 모델은 `SKILL.md`와
`references/`를 읽으며, 이 README는 그 위에 놓인 지도입니다.

---

## 1. 무엇인가

어떤 청중을 대상으로든, 어떤 언어로든, 템플릿이 있든 없든, 출처 자료가 있든 없든 발표용 수준의
`.pptx` 덱을 만들고, 다시 디자인하고, 비평하는 스킬입니다. 이 스킬의 단 하나의 신념은 다음과
같습니다. **슬라이드는 읽기 위한 문서가 아니라 발표자를 위한 시각 보조물이다** — 따라서 모든
결정은 "몇 초 안에 이해된다"는 목표에 최적화됩니다.

이 스킬은 의도적으로 **인터뷰 우선(interview-first)**이며 **비평가 통과제(critic-gated)**입니다.
가정하기 전에 먼저 묻고, 자신의 결과물을 신뢰하지 않습니다 — 덱이 "완성"되었다고 보려면 독립적인
비평가의 동의가 반드시 필요합니다.

---

## 2. 핵심 파이프라인 (auto 모드)

모든 덱은 일곱 단계를 거칩니다 (`SKILL.md`가 권위 있는 명세입니다).

| 단계 | 무엇이 일어나는가 | 왜 존재하는가 |
|---|---|---|
| **0 — Interview** | 하나의 `AskUserQuestion` 묶음: 템플릿, 목적 및 청중, 출처 자료, 스타일. (+후속 질문: 콘퍼런스 장소, 새 템플릿.) | 사용자의 요구사항이 진실의 원천입니다. 그것을 직접 *알아내야* 하며, 이전 덱에서 물려받아서는 안 됩니다. |
| **1 — Understand** | 모든 출처를 깊이 읽고 **이해 브리프(comprehension brief)**를 작성합니다 (한 문장 메시지, 기여, 방법의 본질, 각 그림/표가 *무엇을 위한* 것인지, 한계). | 보기에는 맞지만 작업을 잘못 읽은 덱은 어떤 전문가도 속이지 못합니다. 충실성은 여기서 시작됩니다. |
| **2 — Canvas** | 출력 폴더(`~/Downloads/<deck>/`)를 정하고, 템플릿을 로드하거나 목적에 맞는 룩을 디자인합니다. 팔레트/폰트(CJK용 `EAFONT` 포함)를 설정합니다. | 브랜딩은 레이아웃에 깃들어 있습니다. 디자인은 한 단어가 읽히기 전에 그 문서가 어떤 *종류*인지를 알려야 합니다. |
| **3 — Plan** | 슬라이드 수는 시간 예산에 비례합니다(분당 약 1장): 짧은 발표는 약 6–9장, 더 긴 발표/강의/디펜스/잡토크는 약 10–20장 이상. 각 슬라이드에 하나의 아이디어, 핵심 결론 우선, 목적에 맞게 흐름을 구성. 약 15장 이상 → 섹션 팬아웃(4단계). | 개요를 고치는 것은 싸지만, 완성된 덱을 고치는 것은 비쌉니다. |
| **4 — Build** | `deckkit` 헬퍼를 사용하는 하나의 빌드 스크립트. 전체 출처 그림, 여백(gutter), 회전하는 강조색, 실제 수식, 단일 언어, 선택적 빌드/애니메이션, 발표자 노트. | python-pptx는 빠릅니다. 한 번의 스크립트 실행, 하나의 일관된 저자. |
| **5 — Render + critic loop** | PNG로 렌더링하여 *직접 봅니다*. 그런 다음 **독립적인 비평가 서브에이전트**가 JSON을 반환합니다(동의 / 수정 + 슬라이드별 수정사항). 동의가 나올 때까지 반복합니다. | python-pptx는 보지 않고 씁니다 — 오버플로/대비/글리프 버그는 픽셀에서만 드러납니다. 당신은 자기 작업의 심판이 아닙니다. |
| **6 — Hand off + iterate** | 사용자에게 보여주고, 폴더 경로를 제공하며, 편집 가능성 + 두 가지 변경 레인을 설명하고, 피드백을 반영합니다. | 덱은 사용자의 것이며 계속 안전하게 손볼 수 있어야 합니다. |

**액터-비평가(actor–critic) 루프가 품질 엔진입니다.** 그 *비중*은 중요도에 따라 달라지지만
(랩 미팅에는 비평가 한 명, 콘퍼런스/디펜스/피치에는 서로 다른 관점을 가진 2–3명의 병렬 비평가
패널), 루프 자체는 협상 불가능합니다.

---

## 3. 두 가지 모드

- **Auto (기본):** 인터뷰 → 빌드 → 높은 기준에 이르는 비평가 루프 → 보여주기. 비평가는
  *품질*을 포착합니다.
- **Collaborative (선택 적용):** 저렴한 **승인 게이트**를 추가합니다 — *방향*을 고르고(실제로
  렌더링된 2–3개의 아키타입 슬라이드) → *개요*를 승인하고 → 나머지를 빌드합니다. 게이트는
  비평가가 읽을 수 없는 *선호도*(취향)를 포착합니다. 그 아래의 엔진은 동일하며, 승인만
  추가될 뿐입니다. (`references/collaborative-mode.md`, `scripts/archetypes.py`.)

---

## 4. 시나리오 맵 — 요청이 어떤 경로를 타는가

인터뷰(0단계, 특히 Q3)가 요청의 경로를 결정합니다.

| 사용자가 원하는 것… | 경로 |
|---|---|
| 자신의 코드/논문/문서로 만든 덱 | Build 경로(1–6단계), 콘텐츠 분기 |
| 자료가 없는 덱 | Build 경로. 전문 지식을 바탕으로 초안을 작성하고 웹 검색으로 근거를 다진 뒤 개요를 확정 |
| **자신의** 덱을 **개선**하기 | **Redesign 경로** — 먼저 진단하고, 범위를 확정한 뒤, 사용자의 콘텐츠/그림을 재사용해 재구축 (`references/redesign-existing-deck.md`) |
| **예시처럼 보이는** 덱 | 스타일 모방 — 스타일 브리프를 작성하고 그 룩을 재현 (`references/style-analysis.md`) |
| **콘퍼런스** 발표 | 장소를 식별하고 웹 리서치(규칙, 템플릿, 청중)한 뒤 그에 맞춰 빌드 |
| **포스터** | 범위 한정: 하나의 큰 캔버스. 제작 원칙은 유지되지만 이 스킬은 발표에 맞춰져 있음 — 먼저 사양을 확정 |
| **비영어/CJK** 덱 | `EAFONT` 설정, 단일 언어 규율, CJK 타이포그래피 (`references/multilingual.md`) |
| **큰** 덱(15장 이상) | 선택적 섹션 팬아웃: 공유 `style.py`, 병렬 섹션 저자, `assemble.py`, 비평가 패널 (`references/large-deck-orchestration.md`) |
| **먼저 옵션을 보기** | Collaborative 모드 게이트 |
| **전달 후 변경** | 안전하게 반복 — 수작업 편집을 절대 덮어쓰지 않음 (`references/handoff-and-iteration.md`) |

## 5. 파일 맵

**중추(Spine)**
- `SKILL.md` — 모델이 따르는 운영 명령(0–6단계, 규칙).

**엔진(`scripts/`)**
- `deckkit.py` — 빌드 키트: 텍스트/도형/컴포넌트 헬퍼(`bullet`, `callout`,
  `chip`, `arrow`, `modbox`, `hrule`), 수식(`eq_par`, `equation_png`),
  `speaker_notes`, 대비 검사, 팔레트/폰트(CJK `EAFONT` 포함), 템플릿 재사용
  (`open_template`, `content_slide`), 무템플릿 크롬(`blank_deck`,
  `title_bar`, `footer`). 이를 import하고, 프리미티브를 다시 만들지 마세요.
- `render_deck.sh` — `.pptx` → 슬라이드당 PNG 하나(LibreOffice → PDF → PNG). 크로스 플랫폼.
  병렬/공존 렌더링이 충돌하지 않도록 전용 LibreOffice 프로필을 사용합니다.
- `check_env.sh` — 툴체인을 위한 1회성 사전 점검.
- `anim.py` — python-pptx가 쓸 수 없는 PowerPoint 빌드/애니메이션 타이밍 XML을 주입합니다.
- `assemble.py` — 병렬로 작성된 섹션 모듈을 하나의 덱으로 결합(취약한 병합 없이).
- `archetypes.py` — collaborative 게이트를 위해 방향별로 동일한 미리보기 슬라이드를 빌드.
- `inspect_template.py` — 템플릿의 레이아웃/플레이스홀더/로고를 출력.
- `extract_deck.py` — 기존 덱에서 텍스트/표/그림을 *추출*(재디자인 + 대조).
- `export_notes.py` — 덱의 발표자 노트를 일반 텍스트 리허설 스크립트로 내보내기.

**판단(Judgement)**
- `agents/critic.md` — 독립 비평가의 브리프 + JSON 스키마.
- `references/review-rubrics.md` — 범용 루브릭 + 목적별 오버레이(연구 기반).
- `references/design-principles.md` — 제작 기법과 그 "이유".

**시나리오별 참조**
- `design-by-purpose.md` · `animation.md` · `multilingual.md` · `font-guidance.md` ·
  `style-analysis.md` · `redesign-existing-deck.md` · `collaborative-mode.md` ·
  `large-deck-orchestration.md` · `handoff-and-iteration.md`
- `examples/` — 완성된 빌드 스크립트, 공유 스타일 + 섹션 모듈 규약.

**외부(스킬의 일부가 아님)**
- `~/.claude/slide-templates/` — 사용자의 개인 템플릿 레지스트리. 선택을 위해 읽고,
  새 프로필을 여기에 씁니다. 신규 사용자에게는 비어 있습니다.

## 6. 스킬에 내재된 설계 원칙

1. **아티팩트보다 요구사항.** 템플릿, 예전 덱, 또는 모델의 취향은 *입력*이지 명령이 아닙니다.
   이들이 명시된 요구사항과 충돌하면 요구사항이 이깁니다.
2. **엄격한 충실성.** 모든 주장/수치/그림은 출처로 추적됩니다. 유일한 예외는 명확히
   표시된 미래지향적 내용입니다.
3. **독립적 비평.** 별도의 에이전트가 렌더링된 픽셀을 심판합니다 — 그 독립성이 "동의"를
   의미 있게 만드는 것입니다.
4. **수집은 병렬화하되, 이해는 절대 병렬화하지 않는다.** 읽기/에셋 준비는 팬아웃하되,
   하나의 정신이 일관된 흐름을 붙들어야 합니다.
5. **목적에 맞는 디자인.** 디펜스, 임원 보고, 강의는 서로 같아 보여서는 안 됩니다.
6. **하나의 언어를, 끝까지 일관되게.**
7. **스크립트가 진실의 원천이고, `.pptx`는 아티팩트다.** 재현 가능하며, 사용자의
   편집을 잃지 않고 안전하게 반복할 수 있습니다.

## 7. 알려진 한계 (이에 대해 솔직할 것)

- **텍스트 높이는 측정값이 아니라 추정값이다.** python-pptx는 렌더링된 텍스트 높이를 알 수
  없으므로 `bullet`/`callout` 높이는 비례 추정값입니다 — 오버플로는 렌더 루프(5단계)에서
  잡힙니다. 항상 PNG를 직접 보세요.
- **애니메이션은 정적으로 미리보기할 수 없다.** 렌더는 최종 빌드 상태만 보여줍니다. 빌드
  *순서*는 실제 PowerPoint에서 검증되어 전달 시 사용자에게 설명됩니다.
- **RTL 스크립트(아랍어/히브리어)**는 알려진 약점입니다 — bidi 재배치가 없습니다.
- **포스터**는 최소한으로만 지원됩니다. 이 스킬은 발표에 맞춰져 있습니다.
- **폰트는 임베드되지 않는다**(python-pptx 한계) — 비표준/CJK 폰트 의존성은 전달 시
  반드시 표시하세요.

---

## 8. 툴체인

`python-pptx`, `pymupdf`(렌더), `matplotlib` + `Pillow`(수식/차트), 그리고 렌더링을 위한
LibreOffice(`soffice`). 새 머신에서는 `bash scripts/check_env.sh`를 한 번 실행하세요.
누락된 것이 있으면 정확한 해결책을 출력해 줍니다.
